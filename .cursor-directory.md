## Short description

Manage VMware vCenter 8.0 from Cursor — inventory, snapshots, vMotion,
DRS/HA, vLCM remediation, and the full VM/host/cluster lifecycle, with
built-in dry-run safety and a global read-only kill switch.

## Long description

A Model Context Protocol server that fronts three vSphere API surfaces
(Automation REST, VI/JSON, and SOAP) behind a single session, exposing
the full vCenter management surface to Cursor and other MCP clients.

Features:

- End-to-end VM lifecycle: list, get, create, clone, reconfigure, power
  on/off/reset/suspend, guest-OS-aware shutdown/reboot, vMotion, Storage
  vMotion, console-ticket minting.
- Snapshots: list, create, revert, remove, removeAll.
- Hosts: list, get, enter/exit maintenance, reboot, shutdown,
  disconnect/reconnect, add to cluster.
- Clusters & DRS/HA: create/delete clusters, set DRS and HA policies,
  list and apply DRS recommendations.
- Datastores with recursive glob search across `*.vmdk`, `*.iso`,
  `*.vmx`, ... plus delete and move operations.
- Networks: list, distributed switches and portgroups,
  create/delete portgroups.
- Resource pools, templates, content libraries (incl. publishing), tags.
- Alarms, events, performance counters and queries.
- ISO mount/unmount, customization specs apply.
- Identity & RBAC: roles, permissions, identity providers.
- vSphere Lifecycle Manager (vLCM): cluster-image listing, compliance
  checks, remediation.
- SOAP escape hatch (`soap_runCommand`) for any vim25 method not
  exposed by the modern APIs.

## Permission level

**HIGH** — this server can power off hosts, delete VMs, vMotion across
hosts, remediate clusters with vLCM, and (via the SOAP escape hatch)
invoke arbitrary vim25 methods.

### Safety story

Two layers of safety are built in:

1. **Per-tool dry-run.** Every destructive tool (`*_delete`,
   `*_powerOff`, `snapshot_remove`, `host_reboot`, ...) requires an
   explicit `confirm: true`. Without it, the tool returns a structured
   dry-run preview describing the exact request that would have been
   sent — but never touches vCenter.
2. **Global read-only kill switch.** `VCENTER_READ_ONLY=true` blocks
   every destructive tool regardless of `confirm`. Recommended for any
   agent loop that has not been explicitly authorized to write.

Operators retain full control: the MCP never escalates beyond the
credentials supplied via `VCENTER_USER` / `VCENTER_PASS`, never persists
credentials, and writes logs only to stderr (stdout is reserved for
JSON-RPC and never carries sensitive data).

## Install snippet

```bash
git clone https://github.com/TheEvalon/vmware-vcenter-mcp.git
cd vmware-vcenter-mcp
npm install
npm run build
cp .env.example .env   # then edit with your vCenter details
```

Then register with Cursor (`%USERPROFILE%\.cursor\mcp.json` on Windows,
`~/.cursor/mcp.json` elsewhere):

```json
{
  "mcpServers": {
    "vmware": {
      "command": "node",
      "args": ["<absolute-path-to-clone>/dist/index.js"],
      "env": {
        "VCENTER_HOST": "vcsa.lab.local",
        "VCENTER_USER": "administrator@vsphere.local",
        "VCENTER_PASS": "********",
        "VCENTER_INSECURE": "false",
        "VCENTER_READ_ONLY": "true"
      }
    }
  }
}
```

## Required runtime

- Node.js >= 20
- VMware vCenter Server 8.0 (8.0 U1+ recommended for VI/JSON parity).

Documented in full in the [README](README.md#environment-variables).
